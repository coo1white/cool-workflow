# DESIGN PHILOSOPHY — FREEBSD / UNIX DISCIPLINE

Apply each as a HARD constraint, tied to the real subsystem it governs:

- **Mechanism, not policy.** The driver layer is mechanism: ONE `ExecutionBackend` contract, many interchangeable drivers. v0.1.34 adds *real* HOW/WHERE inside the existing container/remote/ci drivers; it does NOT teach the kernel (orchestrator / dispatch / pipeline-runner) which backend ran. WHAT to run + which evidence to record stays kernel policy.
- **The sandbox profile is the contract — `attestSandbox` is unchanged.** Each real driver maps the resolved profile's five dimensions (`src/sandbox-profile.ts`) onto its real enforcement (container: mounts/dropped-caps/net-namespace/filtered-env; remote/ci: enforced at the runner) and ATTESTS exactly what it enforced via the SAME `SandboxAttestation` shape. A real container that genuinely enforces `read/write/network` keeps the `enforce` it already declares; remote/ci keep `attest` for those dimensions. No new attestation states.
- **Fail closed — the load-bearing rule for THIS feature.** Refusal is the default at every ambiguity: no runtime on PATH (docker/podman), no endpoint (`CW_REMOTE_ENDPOINT`/`CW_CI_ENDPOINT` / `--endpoint`), no image (`--image`/`CW_CONTAINER_IMAGE`), no job target, missing credentials, an unreadable exit status, or a non-zero/timed-out delegation — ANY of these returns a `status:\"refused\"` `ExecutionResultEnvelope` with a stable `refused:<code>` evidence line. The driver NEVER silently downgrades to local/unsandboxed execution. Today the delegating drivers already refuse when no target is configured (`delegationHandle` → `delegation-target-missing`; note the `ci` driver refuses only when BOTH endpoint AND job are absent — either one configures a target); v0.1.34 must preserve that exact refusal envelope when the real path is unavailable, and additionally refuse on real execution failure.
- **Reuse existing records/types — do NOT fork.** Real drivers emit the EXISTING `ExecutionResultEnvelope` / `ResultEnvelope` / `SandboxAttestation` / `BackendExecutionHandle` / `BackendProbeResult` from `src/types.ts`. The `evidence[]` of a real run is the SAME shape the local `executeLocal` path already produces (`command:`, `exitCode:`, `stdoutSha256:`), plus the existing delegation provenance lines (`delegated:`, `handle:`). No new envelope type, no parallel record store.
- **CLI↔MCP parity via the capability-registry (v0.1.27).** Any new verb is declared ONCE in `src/capability-registry.ts` and rendered on both surfaces; `cw <cmd> --json` is byte-identical to `cw_<tool>`. Fail closed on drift (`scripts/parity-check.js`). Prefer extending the existing `backend.probe`/`backend.show` payloads (add a `runtimeAvailable`/`endpointConfigured`/`credentialsPresent` check) over inventing surface-asymmetric verbs.
- **Backward compatible + additive.** Default backend stays `node`; the dispatch path stays a `delegate-host` execution reproducing pre-v0.1.29 behavior exactly. With NO container runtime, NO endpoint, NO credentials present, every existing test and probe output is unchanged. Real execution is strictly opt-in (explicit backend selection + a configured/available target).
- **Deterministic / replayable.** The recorded envelope is byte-stable across backends for the same task; provenance (backendId, handle ref, attestation, ISO timestamps) is recorded AS provenance and is the ONLY part allowed to differ. Only ISO timestamps may be now-derived; a now-derived number fails the payload-identity probe. Eval/replay must compare a container-backed run against a node-backed run and see identical `result`/`evidence`.
- **CW DELEGATES, it does not become the executor.** The real container driver shells out to `docker`/`podman` exec; the remote driver POSTs to a runner endpoint and polls a job; the CI driver triggers a CI job and reads its result. CW captures verifiable evidence (exit + output digest + handle) and records it. CW NEVER reimplements a container runtime, a job scheduler, or a CI system.
- **Not done until documented + tested (`.7.md` voice).** A capability without `docs/real-execution-backends.7.md`, a smoke test wired into `npm test`, and green gates does not exist.

# THE FEATURE — Real Execution Backend Integrations

**CURRENT CODE REALITY.** `src/execution-backend.ts` defines six drivers in `DRIVER_SPECS`. The three local drivers (`node`/`bun`/`shell`, `kind:\"local\"`, `readiness:\"ready\"`) really execute via `executeLocal` (`spawnSync` → exit code + `sha256` stdout digest → canonical envelope; non-zero exit yields `status:\"failed\"`). The three delegating drivers (`container`/`remote`/`ci`, `kind:\"delegating\"`, `readiness:\"unverified\"`) go through `delegate()`, which builds a `BackendExecutionHandle` from `delegationHandle()` (image@digest / endpoint#job) and records a handle + `attestSandbox` attestation + a `status:\"completed\"` envelope WITHOUT EVER EXECUTING ANYTHING — it is a contract-conformant stub. `probeBackend` already checks for `docker`/`podman` on PATH and for `CW_REMOTE_ENDPOINT`/`CW_CI_ENDPOINT`, but only to set `readiness`; nothing consumes a real result. When no target is configured the drivers fail closed (`delegation-target-missing`). `worker-isolation.ts` records `backendId`/`backendSelection`/`backendAttestation` onto the worker scope/manifest and a `worker.backend` trust-audit event, but the dispatch path is always `delegate-host` (host runs the worker), so no backend ever actually executes a worker.

**THE GAP.** The delegating drivers record provenance but do not *do* anything. v0.1.34 makes `delegate()` (or a new focused module it calls) ACTUALLY drive the target — opt-in, fail-closed, byte-stable evidence — for `command`-bearing execution requests, while leaving the contract, types, refusal behavior, and node/bun/shell paths untouched.

Sub-capabilities:

1. **Real container exec.** When `container` is selected AND a runtime (docker/podman) is on PATH AND an image is supplied, run the request's `command`/`args` via `docker run`/`podman run` with the sandbox profile mapped to real flags: read/write paths → mounts (ro/rw), `command`/`env` → entrypoint + filtered `-e`, `network.mode:\"none\"` → `--network none`, dropped capabilities. Capture exit + stdout digest into the SAME envelope `executeLocal` produces, with the container handle in provenance.
2. **Real remote runner.** When `remote` is selected AND an endpoint is configured AND credentials are present, submit the job to the endpoint, poll to completion, and fold the returned exit + output digest into the canonical envelope with the remote handle (`endpoint#jobId`). Enforce a timeout; refuse on transport/auth failure.
3. **Real CI job.** When `ci` is selected AND a CI endpoint/job target is configured, trigger the job, await terminal status, and record exit + output digest + the CI handle into the canonical envelope.
4. **Hardened readiness probe.** Extend `probeBackend` so the existing checks ALSO report credential presence and reachability intent (without performing side-effectful network calls during a read-only probe) so `backend probe` honestly distinguishes `ready` (target present) from `unverified`/`unavailable`. Keep the probe deterministic.
5. **Uniform fail-closed envelope on real failure.** A real exec that errors, times out, returns non-zero, or yields an unparseable result returns a `status:\"refused\"` (or `status:\"failed\"` for clean non-zero, matching `executeLocal`'s semantics) envelope with a stable code — never a fabricated `completed`.
6. **Byte-stable evidence parity.** The `result`/`evidence` of a real container/remote/ci run for a given command are identical to the `node` run of the same command after stripping provenance; an eval/replay comparison and the verifier treat them identically.
7. **Opt-in configuration surface.** A single, documented set of env vars / flags (`CW_CONTAINER_IMAGE`/`--image`, `CW_REMOTE_ENDPOINT`/`--endpoint`, `CW_REMOTE_TOKEN`, `CW_CI_ENDPOINT`/`--job`, etc.) drives selection; absence of any required input fails closed, never defaults to a real call.
