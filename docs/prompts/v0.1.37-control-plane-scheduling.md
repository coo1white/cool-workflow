# THE FEATURE â€” Control-Plane Scheduling

**Current code reality.** `src/run-registry.ts` already owns a durable, ordered, cross-repo queue: `queueAdd` writes `RunQueueEntry` records into `$CW_HOME/registry/queue.json` with a `priority` (default `100` from `DEFAULT_RUN_REGISTRY_POLICY.defaultQueuePriority`), `queueList` returns them sorted by `compareQueue` (priority asc, then `enqueuedAt`, then `id`), and `queueDrain` marks the next N `pending|ready` entries `drained` â€” "CW records readiness/order; the HOST still executes the workers." `RunQueueEntry` (`src/types.ts:2248`) has `status: "pending"|"ready"|"draining"|"drained"|"cancelled"` and no notion of attempts, leases, or concurrency. `RunRegistryPolicy` (`src/types.ts:2267`) carries only retention + `defaultQueuePriority`. The `Scheduler`/daemon in `src/scheduler.ts` + `src/daemon.ts` is an unrelated wall-clock/cron *reminder* scheduler (loop/cron/reminder tasks), NOT a control-plane run scheduler. `src/triggers.ts` fires routine events. The capability registry already declares `queue.add|list|drain|show` (`src/capability-registry.ts:382`).

**The precise GAP.** The queue has ORDER but no **scheduling policy**: nothing limits how many runs are in flight (no concurrency ceiling), nothing retries a transient failure with backoff, and nothing bounds retries â€” `queueDrain` would happily re-hand the same failing entry forever. There is no *lease* concept (a claimed, time-bounded selection the host holds while executing) and no *park* state (a terminal "give up, needs operator" state distinct from `cancelled`). v0.1.37 closes this by layering policy-as-data over the existing queue.

Sub-capabilities:

1. **Priority + readiness selection.** A pure `selectLeases(policy, queue, now)` that returns the next eligible entries in deterministic order (reuse `compareQueue`), skipping anything not yet eligible (`nextEligibleAt > now`), parked, or already leased.
2. **Concurrency limit as a hard ceiling.** `policy.maxConcurrent` bounds in-flight (`leased`) entries per scope (and optionally per-app/per-repo class). Leasing stops at the ceiling; over-limit entries stay `ready`. Never exceed it.
3. **Leases.** Claim an entry â†’ `status: "leased"` with a `leaseId` and `leaseExpiresAt` (TTL from policy). `leaseRelease`/`leaseComplete` returns it to terminal or to retry; an EXPIRED lease is reclaimable (the host died) and counts as one failed attempt â€” recorded, not silently reset.
4. **Retry with computed backoff.** On a failed/expired attempt under budget, increment `attempts`, set `nextEligibleAt = now + backoff(attempts)` using a deterministic policy curve (e.g. `baseMs * factor^(attempts-1)`, capped), set `status` back to `ready`. No randomness in the core (jitter, if ever, is a separate declared knob).
5. **Park past budget (fail closed).** When `attempts >= policy.maxAttempts`, set `status: "parked"` with a `parkedReason`; it is never re-selected. An explicit `schedule reset <id>` (operator action) is the ONLY way back to `ready`.
6. **A `schedule plan` read-only verb.** Returns the would-be lease plan for the current queue+policy+now WITHOUT mutating â€” deterministic, replayable, payload-identical across surfaces, embeddable in the Workbench.
7. **Policy get/set as data.** `schedule policy show|set` over a durable `SchedulingPolicy` record (concurrency, maxAttempts, backoff curve, lease TTL), defaulting to the conservative fail-closed values when absent.
