# ROLE

You are a senior runtime engineer on Cool Workflow (CW) — a TypeScript/Node agent workflow control-plane shipped as a Claude/Codex plugin (cw_* MCP tools + a `node scripts/cw.js` CLI), repo currently at v0.1.38. CW's discipline is host-mechanism / policy-as-data: the per-run `.cw/runs/<id>/state.json` is the single source of truth, and every higher layer is a DERIVED, rebuildable, fail-closed view over it. CW is the auditable orchestration / control-plane layer — it deliberately does NOT execute the model; it makes someone else's execution explicit, inspectable, recoverable, and replayable (`DIRECTION.md`). You are building on a precise lineage: v0.1.28 added the Run Registry + a durable run queue and the **archive overlay** (`run-registry.ts` — "Archive is an overlay mark, not a delete"); v0.1.32 added append-only collaboration (attested approvals/comments/handoffs); v0.1.35 added per-node snapshot/diff/deterministic replay (`node-snapshot.ts`: `snapshotNode` / `replayNodeSnapshot` / `verifyNodeReplay`); v0.1.37 added control-plane scheduling as policy-as-data (`sched policy`, lease TTL, `reclaimExpired`).

Your v0.1.38→v0.1.39 task: give CW a **provable reclamation** mechanism that reclaims disk WITHOUT violating the audit/replay moat. Today this is a live problem — a single day of dogfooding produced **~1 GB across 200+ runs** under `.cw/runs/`, and there is currently **zero disk reclamation**: `run archive` only marks an overlay (never frees bytes), `sched reclaim` reclaims expired leases (not disk), and worker scratch dirs are never cleaned. Naive GC is forbidden because CW's entire value is "don't trust, verify." So reclamation must be a **verifiable, append-only state transition**: freeing bytes leaves behind cryptographic proof that what was freed is reconstructable-or-worthless and that the audit-essential subset is sealed.

# DESIGN PHILOSOPHY — FREEBSD / UNIX DISCIPLINE

Each tenet is a hard constraint; load-bearing ones are flagged.

- **THE INVARIANT — never delete what is audit-essential AND irreproducible [LOAD-BEARING].** Reclamation may free only two classes of bytes: (1) artifacts deterministically reconstructable from retained inputs + a recorded recipe + an `expectDigest`, or (2) pure scratch with zero audit value. A byte is freeable ONLY if it is also **referenced by no surviving evidence locator or audit/collaboration event**. This is the hard rule for `blackboard/` and `candidates/`: a blackboard artifact can BE an evidence locator (`recordTrustAuditEvent` carries `blackboardArtifactRefId`/`blackboardMessageId`; `recordWorkerOutput` publishes blackboard refs as `evidenceRefs`), so the per-kind classifier MUST walk the evidence/audit reference graph and **fail closed (free nothing for that path) when any reference is found** — a referenced blackboard digest forces retention, or recipe+`expectDigest` capture if reconstructable. The **skeleton schema is the contract** for what must survive every reclamation (export `SKELETON_REQUIRED_KEYS` / a `validateSkeleton()` so completeness is machine-checkable): final verdict, commit record, every evidence locator's content digest, the attestation chain, the cost record, and the append-only audit + collaboration (v0.1.32) logs. If extracting a complete skeleton fails, reclamation **fails closed** and frees nothing.
- **Write-ahead, fail-closed sequencing — order is the safety property [LOAD-BEARING].** The reclamation transaction MUST be: (1) extract + seal skeleton → (2) write the full tombstone (freed-manifest with a pre-deletion sha256 per path) → (3) fsync/commit the tombstone into the append-only registry → (4) ONLY THEN free the bulk bytes. A crash between any steps must leave EITHER the full run OR a complete tombstone — never a half-deleted run with no proof. Make this **testable by design**: expose the transaction as discrete, individually-callable steps (`extractSkeleton()`, `buildTombstone()`, `commitTombstone()`, `freeBulk()`) plus a `runReclamation(run, policy, { faultAfter?: "skeleton"|"tombstone-write"|"tombstone-commit" })` that throws a synthetic `ReclamationAbort` after the named step — never by killing the process.
- **Reclamation is append-only — it EXTENDS history, never rewrites it [LOAD-BEARING].** The tombstone is written as a NEW `reclaimed.json` overlay (peer of `archive.json`); **only the bulk DATA bytes are freed — no existing audit, state, or commit record is ever rewritten.** It is itself a new audit record, hash-chained: `tombstoneHash` is recomputed from the freed-manifest + sealed skeleton, `prevTombstoneHash` = the prior reclaimed tombstone's `tombstoneHash` within the run (genesis = sha256 of the sealed skeleton), so a tampered registry entry is caught. `gc verify` MUST recompute `tombstoneHash` independently from the freed-manifest + skeleton, never trust the stored value. Record the attestation through the existing append-only `recordTrustAuditEvent` (`fs.appendFileSync`), not a new mutable store. `run archive` keeps its mark-only semantics untouched; `reclaim` is the NEW disk-freeing tier and still leaves a tombstone. Nothing in the original audit log is ever edited or erased.
- **Capability downgrade is explicit and queryable — never silent [LOAD-BEARING for replay].** Reclaiming a node snapshot (v0.1.35) downgrades a run from `re-runnable` to `verify-only` (or `re-runnable-by-reconstruction` when inputs + `expectDigest` are retained). `cw run show <id>` MUST report `record.tier`, `record.capability`, and an enumerable `record.capabilityReason` (closed set, e.g. `snapshot-reclaimed-no-reconstruction` | `inputs-and-expectdigest-retained`) — assert the exact enum, never free-text prose. **Reconstruction is a distinct code path, NOT live `verifyNodeReplay`:** `verifyNodeReplay`/`snapshotNode` re-read the LIVE source node + its artifact paths, which reclamation has freed — so for a reclaimed run, `loadNodeSnapshot` returning `absent` on a freed artifact is the EXPECTED, correct fail-closed signal. The reconstruction verifier instead re-runs the **recorded recipe against the RETAINED inputs** (keyed on the retained-inputs digest) and compares the result's sha256 to the tombstoned `expectDigest`; it must never route through the freed source bytes.
- **No surviving node may reference a freed path [LOAD-BEARING for replay].** Before freeing worker scratch, the result node's `worker-result` artifact — which `recordWorkerOutput` sets to `absoluteResultPath` INSIDE the scratch dir — MUST be re-pointed to the retained `results/<task-id>.md` copy (or the artifact dropped after sealing its pre-deletion sha256 into the tombstone). Otherwise the result node's snapshot dangles and `loadNodeSnapshot` goes `absent` — a real, silent replay downgrade. Eager scratch reclaim is a transaction precondition: post-reclaim `loadNodeSnapshot` of the result node MUST be `valid` (not `absent`), proven via `verifyNodeReplay` against the rewritten artifact set, before the free is allowed.
- **Content-addressing is the faithfulness proof.** Every freed path's sha256 is recorded in the tombstone BEFORE deletion, so even with the bytes gone you can prove exactly what existed and verify any later-presented copy against it.
- **Policy-as-data, fail-closed, dry-run-first.** Retention/reclamation thresholds are policy — they extend `RunRegistryPolicy` (`src/types/run-registry.ts`, alongside `archiveOlderThanDays`/`archiveStates`), NOT `SchedulingPolicy` and NOT a new policy file — never baked in. `cw gc plan` is a pure dry-run that frees nothing. Auto-reclamation is opt-in and bounded; **CW never deletes by default.** The single carved exception is worker scratch: `kind: scratch` with zero audit value, its digests tombstoned and `result.md` already evidence-gated under `results/`, is the ONE class reclaimed eagerly-by-default (opt-out `--keep-scratch`); everything else is opt-in.

# THE FEATURE — Run Retention & Provable Reclamation

A new lifecycle tier and a small, verifiable GC built on the existing archive overlay.

**Tiers (each transition recorded, policy-gated, fail-closed):**

```
live      full on disk            re-runnable + verifiable
archived  overlay mark, full bytes  re-runnable + verifiable   ← current ceiling (v0.1.28)
reclaimed tombstone + skeleton + digests  verify-only (or re-runnable-by-reconstruction)  ← NEW (v0.1.39 ceiling)
```

The lifecycle ceiling for this release is `reclaimed`. (A future `forgotten` compliance tier — discarding even the skeleton, keeping only the chained tombstone hash — is out of scope here; the `reclaimed` hash chain is DESIGNED to extend to it later. See NON-GOALS.)

**Sub-capabilities:**

1. **`cw gc plan` (dry-run)** — given a policy (or `--olderThanDays` / `--runId`), compute eligible runs (predicate below), the exact bytes that WOULD be freed per kind (scratch / reconstructable-snapshot / candidate / reference-free blackboard / …), and the per-run capability downgrade. Frees nothing. JSON + human output. **Eligibility is explicit and fail-closed:** `derivedLifecycle ∈ {completed, failed}` AND `record.archived === true` AND no open feedback AND past `reclaimAfterArchiveDays` (mirror `archiveByPolicy`'s filter, plus the archived+retention gate). `running`/`blocked`/`queued` are NEVER reclaimable; the check reads live source state and fails closed on unreadable.
2. **`cw gc run` (reclaim)** — execute the write-ahead transaction above for eligible runs, producing the tombstone and freeing the bulk. Bounded (`--limit`, byte/quota ceilings), fail-closed on any incomplete skeleton.
3. **`cw gc verify <id>`** — re-prove a reclaimed run: skeleton schema-complete (every key in `SKELETON_REQUIRED_KEYS`)? `tombstoneHash`/`prevTombstoneHash` chain recomputed-and-untampered? each reconstructable artifact re-derived from its RETAINED inputs (NOT the freed source) to its `expectDigest`? was the run eligible when reclaimed? All pass ⇒ the reclamation is proven correct.
4. **Eager worker-scratch reclaim** — the degenerate instance: a worker's scratch dir is `kind: scratch`, zero audit value, and `result.md` is already copied to `results/<task-id>.md` and evidence-gated. Reclaim it immediately after accept (opt-out via `--keep-scratch`), tombstoning its digests. **Precondition before freeing:** re-point the result node's `worker-result` artifact (set by `recordWorkerOutput` to `absoluteResultPath` inside the scratch dir) to the retained `results/<task-id>.md` copy, so no surviving node references a freed path; prove via `verifyNodeReplay` against the rewritten artifact set that the result-node snapshot stays `valid`. Only then is capability genuinely unchanged.
5. **Policy extension** — add reclamation fields to `RunRegistryPolicy` (NOT `SchedulingPolicy`): `reclaimAfterArchiveDays`, `keepSnapshots`, `keepScratch`, `reclaimStates`, byte/run quotas. Back-compatible defaults that reclaim nothing.

# NON-GOALS — DO NOT (each guards a known creep vector)

- Do NOT delete, edit, or rewrite any audit record, evidence digest, commit record, attestation, or append-only collaboration entry (v0.1.32) — ever. Reclamation only ADDS tombstones.
- Do NOT change `run archive`'s mark-only semantics. Archive ≠ reclaim; they are distinct tiers.
- Do NOT silently downgrade capability. A reclaimed run that lost replayability MUST advertise `verify-only` with a reason.
- Do NOT free any byte before the tombstone is committed (write-ahead is mandatory, not an optimization).
- Do NOT ship a daemon/cron that auto-reclaims by default. Opt-in, dry-run-first, bounded.
- Do NOT reclaim live or non-terminal runs, or runs not yet archived/past-retention. The eligible set is exactly `derivedLifecycle ∈ {completed, failed}` AND archived AND no open feedback AND past retention; everything else refuses.
- Do NOT implement the `forgotten` tier (discarding the skeleton, retaining only the chained tombstone hash) in v0.1.39 — the lifecycle ceiling for this release is `reclaimed`, and the `RunLifecycleState` union gains ONLY `reclaimed`. `forgotten` is a future compliance tier with its own append-only-proof-of-lawful-discard threat model; the `reclaimed` hash chain is designed to extend to it later, but no `forgotten` enum value, verb, or transition ships now.
- Do NOT free any path that is not explicitly classified `reconstructable` or `pure-scratch` in the retention manifest. Any unclassified path defaults to RETAINED (fail-closed). Hard ALLOW-LIST (never freed): `state.json`, `audit/`, `commits/`, the collaboration log, the attestation chain, and the new `reclaimed.json`. `report.md` is treated as derived-and-reproducible-from-state — retain it explicitly, never free it implicitly.
- Do NOT change the kernel `state.json` schema beyond adding the tombstone artifact + policy fields; pre-v0.1.39 runs must load unchanged.
