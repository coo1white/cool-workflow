# DESIGN PHILOSOPHY — FREEBSD / UNIX DISCIPLINE

Each tenet is a hard constraint; the load-bearing one for this feature is flagged.

- **THE RED LINE — delegate (spawn external process) vs internalize (import SDK) [LOAD-BEARING].** `src/execution-backend.ts` states it verbatim: "CW DELEGATES, IT DOES NOT BECOME THE EXECUTOR." The `agent` driver does the SAME thing the `container`/`remote`/`ci` drivers do — it `spawnSync`s an out-of-process child (the agent CLI) or POSTs to a configured HTTP agent endpoint, then records a `BackendExecutionHandle` + `SandboxAttestation` + the canonical result envelope. It **MUST NOT** `import @anthropic-ai/...`, `import openai`, add any model-SDK dependency to `package.json`, hold an API key, construct a chat-completions request, or call a model HTTP API. The model lives in the agent's process; CW spawns it and records the attested output. If you find yourself adding a provider SDK to `dependencies`, STOP — that is the red line and it loses the neutral-audit moat (DIRECTION.md 决策过滤器 #1: "它让别人的执行更可审计，还是让 CW 自己去执行？→ 后者直接否").
- **Operator-chosen model is policy; agent-reported model is the attestation — NEVER conflate them [LOAD-BEARING for attestation integrity].** Any model id CW passes INTO the agent invocation (`CW_AGENT_MODEL` interpolated into the command-template, e.g. `--model {{model}}`) is **policy-as-data the OPERATOR chose**; it is recorded ONLY as part of the secret-stripped command-template/args provenance, and is **NEVER** the source of the attested `UsageRecord.model`. The recorded/attested model id MUST come solely from what the external agent reports back in its result/usage output (today the code populates `UsageRecord.model` only from host-reported `raw.model`, `observability.ts`; CW never synthesizes it). If the agent reports no model, record `unreported` — never backfill from `CW_AGENT_MODEL`. A configured `CW_AGENT_MODEL` that differs from the agent's reported model MUST NOT overwrite the host-reported model id.
- **Fail closed — the exact ambiguous→refusal rule.** The `agent` backend is `kind: "delegating"`, `readiness: "unverified"`, and **fails closed when unconfigured**. Mirror the `remote`/`ci` pattern EXACTLY, distinguishing the two surfaces: (a) the *probe* — `probeBackend("agent")` reports `readiness: "ready"` (`ready: true`) iff a command-template/endpoint is configured, otherwise `readiness: "unverified"` (`ready: false`) with a non-empty human reason — byte-identical in shape to `probeBackend("remote")` unconfigured (`src/execution-backend.ts:353`), NEVER a hard `"refused"`/`"unavailable"`; (b) the *runBackend* path — unconfigured execution (no command-template AND no endpoint) returns a `delegation-target-missing` refusal envelope (`execution-backend.ts:510`), exactly like `remote`/`ci`. A spawned agent that exits non-zero, produces no `result.md`, produces a `result.md` that fails `validateResultEnvelope`, or returns no exitCode ⇒ a `refused`/`failed` envelope, NEVER a fabricated `completed`. In the drive loop, a worker whose agent hop fails exhausts its scheduling retry budget and lands `parked` (reuse v0.1.37 `retryOrPark`) — never silently re-driven forever.
- **Deterministic / replayable, bound to the v0.1.35 engine [LOAD-BEARING for the moat].** The attested record (model id, prompt digest, args, result digest, exit) is plain data; replaying the recorded run reproduces the SAME audit/provenance graph. The agent's *output text* is non-deterministic, so it is captured as a `sha256` **digest** recorded AS provenance (never re-invoked on replay). Bind these digests to the EXISTING v0.1.35 node-snapshot/replay engine: the prompt/result/model digests are part of the snapshotted node body / `sourceFingerprint`+`outputFingerprint` so `snapshotNode`/`replayNodeSnapshot`/`verifyNodeReplay` (`src/node-snapshot.ts`) already cover them and fail closed on drift (`stale`/`absent`). Replay **re-verifies the recorded digests via the existing engine and MUST NOT re-spawn the agent or re-read the live `result.md`** (DIRECTION.md 决策过滤器 #2: replay 确定性不可妥协). The drive loop's *ordering* (which worker next) is the existing deterministic `sched plan` over the queue.
- **Two layers, never conflated — backend evidence triple vs `result.md` acceptance [LOAD-BEARING].** `runAgentProcess` records the backend evidence triple from the agent CHILD's `command` + `exitCode` + `sha256(stdout)` — identical mechanism to `runContainer`/`runHttpDelegation` (`delegatedEnvelope`, `execution-backend.ts:542`); it never reads, parses, or hashes a `result.md` into that triple. The worker's `result.md` `cw:result` envelope is accepted in a SEPARATE layer (`recordWorkerOutput`, `worker-isolation.ts:269`) which writes its own `source: "cw:result"` / `cw-validated-worker-output` evidence — unchanged by this feature. Do NOT fabricate a stdout digest from `result.md` contents and do NOT fold the agent's free-text `result.md` into the backend evidence triple; either corrupts the evidence chain the moat depends on.
- **Reuse existing records/types, don't fork (DIRECTION.md: "把已有资产用得更狠").** The driver is a new row in `DRIVER_SPECS` + a `runAgentProcess` alongside `runContainer`/`runHttpDelegation`, returning the SAME `delegatedEnvelope`. The handle reuses `BackendExecutionHandle` (its `kind` union ALREADY includes `"process"`). The model id reuses the existing host-attested `UsageRecord.model` ("Host-reported model id … CW never synthesizes this", `src/types/observability.ts`). The drive loop wires the EXISTING `plan` / `dispatch` / `worker output` / `commit` chain and the v0.1.37 `sched` scheduler — it introduces no second runner.
- **Mechanism vs policy.** The `agent` backend is mechanism; WHICH agent (claude / codex / ollama / an HTTP endpoint) is **policy expressed as DATA** — a command-template string and/or endpoint resolved from flags > env > a durable config record. claude / codex / ollama are CONFIGS, never CW dependencies.
- **Stable static descriptors — `locality` does not vary at runtime.** `locality`/`kind` are FIXED properties of a `DRIVER_SPECS` row (read statically by `specDescriptor`, stamped into `delegatedEnvelope`/`refusedEnvelope` provenance, `execution-backend.ts:169/298/490`). The `agent` row has ONE fixed `locality` (`local`); an HTTP endpoint is treated as a configured delegation target the same way `remote` already mixes a local Node child posting to a remote endpoint — the recorded provenance `locality` MUST be stable and match the static descriptor for any run, regardless of whether a command-template vs endpoint is configured. Do not let `locality` flip by env.
- **Identical envelopes, any backend.** `runAgentProcess` returns the canonical `command:`/`exitCode:`/`stdoutSha256:` evidence triple `delegatedEnvelope` already produces, so the backend-execution envelope SHAPE is byte-stable across `node`/`container`/`remote`/`ci`/`agent`; the agent handle + model/prompt/result digests live in `provenance`, NEVER in `evidence`.
- **CLI ↔ MCP parity via the capability registry (v0.1.27).** Every new verb (`run --drive` control verbs, agent backend config) is declared ONCE in `src/capability-registry.ts`; `cw <cmd> --json` is byte-identical to `cw_<tool>` for read-only verbs (probed); mutating/surface-specific verbs carry a `reason`. Read-only drive/config payloads MUST be deterministic given a fixed run — **no now-derived numeric fields** (only ISO timestamps may be now-derived; the parity probe strips them, a now-derived number fails the payload-identity gate — per the parity-payload-now-derived rule). Drift fails the release.
- **Backward compatible + additive.** Adding the `agent` row leaves `node`/`bun`/`shell`/`container`/`remote`/`ci` byte-identical; `backendIds()` simply grows by one. A run driven manually (today's plan→dispatch→`worker output`→commit) still works unchanged. No `.cw/` layout change.
- **Not done until documented + tested (.7.md voice).** The feature does not exist until `docs/agent-delegation-drive.7.md` explains the delegate-not-internalize boundary, the drive lifecycle, and the fail-closed rules; the smoke asserts the unconfigured refusal + the no-SDK guard; the dogfood `--smoke` proves the audited-report shape; and all gates are green.

# THE FEATURE — Agent Delegation Drive

**Current code reality.** `architecture-review` (`apps/architecture-review/workflow.js`) defines **13 prompt workers** via `agent(...)` across the Map (×6), Assess (×6) and Verify (×1) phases, PLUS **1 Verdict artifact** via `artifact(...)` — **14 worker scopes total** under the `readonly` sandbox profile, each a natural-language prompt. (Verify IS a real worker id, `verify:p0-p2-risks`; Verdict is the lone `artifact` node, `verdict:synthesis`.) CW can `plan` it, `dispatch` tasks, allocate isolated worker scopes (`worker-isolation.ts` writes `input.md` + a manifest and waits for a `result.md`), and accept output via `recordWorkerOutput` (which validates the `cw:result` envelope, copies it into `resultsDir`, runs the verifier gate, and records trust-audit + provenance). `src/execution-backend.ts` already has the delegating pattern: `runBackend` → `delegate` → `runContainer` (real `docker run`) / `runHttpDelegation` (real POST via a Node child), each returning `delegatedEnvelope` with a `BackendExecutionHandle` in provenance and failing closed (`delegation-target-missing`, `runtime-unavailable`, `delegation-failed`) rather than fabricating a result. v0.1.37 added `sched plan|lease|release|complete|reclaim|reset` over the queue.

**The precise GAP — the manual worker-fulfillment hole.** Nothing SPAWNS the agent that writes each `result.md`. Today, running `architecture-review` end-to-end requires an **external operator to hand-write all 14 `result.md` files** — CW plans, isolates, and accepts, but the actual "go read the repo and answer the prompt" step is a human/agent doing it out-of-band, one file at a time, with no recorded attestation of WHICH agent/model produced each file. There is no `agent` execution backend, and `cw run` is run-**registry** management (`run search|list|show|resume|archive|rerun`), NOT an auto-drive loop. So the turnkey product ("point CW at a repo, get an audited risk report") does not yet exist: the last mile is manual.

**Sub-capabilities:**

1. **An `agent` delegating backend driver.** A new `DRIVER_SPECS` row (`kind: "delegating"`, **fixed** `locality: "local"`, `readiness: "unverified"`, `default: false`) and a `runAgentProcess` alongside `runContainer`/`runHttpDelegation`. It spawns the configured agent **argv-style (`shell: false`, like every existing delegating driver — never a shell-interpreted string)** with `{{...}}` substitution into discrete argv elements (command-template, e.g. `claude -p --output-format json {{manifest}}`), or POSTs the manifest to an agent endpoint, then waits for the worker's `result.md` (accepted by the separate `recordWorkerOutput` layer) and returns the canonical `delegatedEnvelope` whose evidence triple is the agent CHILD's `command`/`exitCode`/`sha256(stdout)` — NOT the `result.md`. Vendor-neutral: the command-template / endpoint / model is resolved flags > env (`CW_AGENT_COMMAND` / `CW_AGENT_ENDPOINT` / `CW_AGENT_MODEL`) > durable config; `CW_AGENT_MODEL` is interpolated into the template as operator policy and recorded only in secret-stripped args provenance, never as the attested model id. **Fails closed when unconfigured** (`delegation-target-missing` on the runBackend path), on non-zero exit, or on a missing/invalid `result.md`. `probeBackend("agent")` reports `readiness: "ready"` iff a template or endpoint is configured, else `readiness: "unverified"` / `ready: false` with a reason (mirror the `remote`/`ci` probe exactly).
2. **A `cw run <app> --drive` auto-advance loop.** Wire the EXISTING `plan` → `dispatch` → agent-fulfill (via the `agent` backend) → `recordWorkerOutput`/verify → `commit` chain into one loop, plugged into the v0.1.37 scheduler: `sched plan` picks the next eligible worker deterministically, the loop leases it, the `agent` backend fulfills it out-of-process, accepted output completes the lease, a failed hop releases-as-failed (retry/backoff/park). The loop is a thin orchestrator over existing verbs — it spawns NOTHING itself except through the `agent` backend, introduces NO second queue/runner/scheduler. The Verdict `artifact` node is fulfilled through the SAME agent backend (it is a worker scope with a `result.md` like any other); state this explicitly so the loop's per-node dispatch and terminal condition are unambiguous. `--drive --once` advances exactly one step (deterministic, inspectable, injected `now`); bare `--drive` runs to completion or to a parked/blocked stop.
3. **Attestation / provenance for the agent hop.** For each agent-fulfilled worker, record AS provenance (not evidence): the host-attested **model id** sourced SOLELY from what the agent reports back (reuse `UsageRecord.model`; record `unreported` if the agent reports none, never backfill from `CW_AGENT_MODEL`), the **prompt digest** (`sha256` of the worker `input.md`/manifest prompt), the **secret-stripped agent args / command-template** (redact any arg matching a configured key pattern or following `--api-key`/`--token`/`--key`; never record raw env values), and the **result digest** (`sha256` of `result.md`). These digests are folded into the snapshotted node body so the v0.1.35 replay engine covers them. Reuse the v0.1.31 observability `UsageRecord` and the `trust-audit` event chain (a new `worker.agent-delegation` audit event hung off the existing `worker.backend` / `worker.output` events) so the run stays auditable & replayable. The `BackendExecutionHandle` is `kind: "process"` with `ref` = the agent invocation reference.
4. **ONE real dogfood (the proof) + a hermetic `--smoke`.** Extend `dogfood-one-real-repo` / `scripts/dogfood-release.js` discipline with an `architecture-review` dogfood. The **live full-drive against ONE real external repository with a real configured agent** (producing the committed audited risk report — the Verdict `result.md` + audit/provenance) is a **MAINTAINER-run, out-of-band proof, explicitly OUT of CI** (mirror the "Promote To Real Release Actions" section of `dogfood-one-real-repo.7.md`). The CI/release gate is a `--smoke` mode (a stub agent, no live binary, no second repo) that exercises the unconfigured-refusal and park paths for `npm test` without recursive agent spawning, mirroring `dogfood-release-smoke.js`.

# NON-GOALS — DO NOT (each guards a known creep vector)

- **No internalized model.** No model SDK in `dependencies`/`devDependencies`, no API key held by CW, no chat/completions request constructed or called. (Restated from the red line; enforced by the no-SDK guard test.)
- **No changes to architecture-review's static phases or worker prompts.** The 4 phases / 14 worker scopes stay STATIC — dynamic/added phases break replay determinism and are an explicitly-flagged bad suggestion. Wire only the drive entry.
- **No re-spawning the agent on replay.** Replay re-verifies the recorded prompt/result/model digests via the existing v0.1.35 engine; it never re-invokes the agent or re-reads the live `result.md`.
- **No second queue / runner / scheduler.** Reuse the v0.1.28 queue and the v0.1.37 scheduling (`scheduling.ts`, `retryOrPark`). The drive loop is a thin orchestrator over existing verbs.
- **NOT the bet-B framework-observability path.** Do not add LangChain/CrewAI/agent-framework ingestion — that is a separate product line, out of scope here.
- **No `.cw/` layout change** and **no new dedicated Workbench surface** (the agent backend surfaces through existing `backend.list/show/probe`; drive state through existing run/worker/audit verbs — defer any dedicated drive panel to a later rendering-focused increment).
