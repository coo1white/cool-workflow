# DESIGN PHILOSOPHY — FreeBSD / Unix discipline

Each tenet is a hard constraint, not a suggestion. The load-bearing rule for THIS feature is flagged.

- **Mechanism, not policy.** The migration registry is mechanism: a declared map of `{ from, to, up, proof }`. WHICH schema is current and WHICH versions are still supported is policy expressed as data (`version.ts` constants, per-contract `compatibility` ranges) — never branching buried in the transform.
- **Fail closed on ambiguity. [LOAD-BEARING FOR THIS FEATURE]** A schema version with no declared path to current is an unsupported error, never a silent pass-through. The exact rule: if `detected < MIN_SUPPORTED`, or `detected > CURRENT`, or no registry edge chains `detected → CURRENT`, or a declared `up` transform drops/overwrites any source record, → `status: "unsupported"` with a named error and NO write. Today `state-migrations.ts` already refuses newer-than-runtime and below-minimum versions and errors on a missing step (line 109, `"No migration step from run-state schemaVersion ..."`); v0.1.36 makes that refusal a first-class registry property checked up front, not a loop fallthrough, and extends it to the workflow-app schema.
- **Reuse existing records and types; do not fork.** Build on `StateMigrationStep`, `StateMigrationReport`, `StateMigrationChange`, and `StateCompatibilityStatus` already in `state-migrations.ts`, and the `compatibility` shape already in `pipeline-contract.ts` / `validateCompatibility` in `workflow-app-sdk.ts`. The registry wraps the existing `migrateRunState` pipeline (detect → migrate → normalize → validate → report); it does not replace it.
- **CLI↔MCP parity via the capability registry (v0.1.27).** Every new verb is declared once in `capability-registry.ts` and rendered identically on both surfaces. A `--json` payload from `cw <cmd>` MUST equal the `cw_<tool>` MCP result byte-for-byte; `npm run parity:check` fails closed on any undeclared tool, command, or payload divergence.
- **Backward compatible and additive.** No run-state schema bump unless a transform genuinely requires it; `CURRENT_RUN_STATE_SCHEMA_VERSION` and `WORKFLOW_APP_SCHEMA_VERSION` stay where they are unless this feature adds a real new shape. Pre-v0.1.36 runs and apps load unchanged; the migration registry is a derived, declared view over the existing pipeline. The whole point of declaring migrations is to make FUTURE bumps safe, not to force one now.
- **Append-only; never overwrite the source snapshot.** The original `state.json` (and app manifest) is the authority. A migration produces a NEW normalized object and a change list; the round-trip prover writes proofs/snapshots beside the run, never over it. This mirrors v0.1.32 collaboration's append-only `supersedes` rule and v0.1.14's "preserve unknown user data by copying" rule — a correction is a new record, not an in-place edit.
- **Deterministic and replayable.** Given the same input state and the same registry, the migration verdict, change list, and compatibility proof are byte-stable (no now-derived numbers; only ISO timestamps may be now-derived, per the parity payload rule). The same fingerprint must reproduce across runs so `fixture-compat` can hash-before/hash-after and prove the source was not mutated.
- **Not done until documented and tested.** A capability without `docs/contract-migration-tooling.7.md`, a smoke test wired into `npm test` / `release:check` / `version:sync`, and an index/README/CHANGELOG entry does not exist. Write the doc in the established `.7(.md)` operator voice (who is affected, the policy, the dry-run default, the unsupported cases).

# THE FEATURE — Contract Migration Tooling

**Current reality.** `src/state-migrations.ts` already versions run-state schema and migrates step-by-step: `RUN_STATE_MIGRATIONS` is a hand-maintained array of `StateMigrationStep { from, to, description, migrate() }`, and `migrateRunState` walks it in a `while (schemaVersion < CURRENT)` loop (lines 105–114), detecting the version, refusing below-minimum and newer-than-runtime states, then normalizing and validating. `src/version.ts` holds the four schema constants (`CURRENT_RUN_STATE_SCHEMA_VERSION = 1`, `LEGACY = 0`, `MIN_SUPPORTED = 0`, plus `WORKFLOW_APP_SCHEMA_VERSION = 1`). The workflow-app schema lives on a SEPARATE, unrelated track: `workflow-app-sdk.ts` checks `app.schemaVersion !== WORKFLOW_APP_SCHEMA_VERSION` and validates a free-text `compatibility` block, but has NO migration path at all (an old app schema is rejected, never migrated). `pipeline-contract.ts` carries a static `compatibility: { minSchemaVersion, maxSchemaVersion, notes }` that nothing enforces as a migration boundary.

**The gap.** Migration is ad-hoc and run-state-only: the step list is an inline array with no declared `up` transform abstraction, no recorded compatibility PROOF that a step is safe, no first-class fail-closed check for an unreachable version, no round-trip/non-destructiveness guarantee, and NOTHING covering the workflow-app schema. v0.1.36 makes migration a first-class, declared subsystem shared by both contracts.

Sub-capabilities:

1. **Declared migration registry.** A typed registry of edges `{ contract: "run-state" | "workflow-app", from, to, description, up(snapshot) → migrated, proof }` for each schema contract. `RUN_STATE_MIGRATIONS` becomes the run-state instance of this registry (reusing `StateMigrationStep`); a parallel registry covers the workflow-app schema. The registry is the single declared source for "what versions exist and how to advance them."
2. **Compatibility proof per edge.** Each edge records an explicit compatibility proof — the invariant it preserves (e.g. "adds defaulted field X, drops nothing," "every source key present in output"). The proof is data, checkable mechanically by the prover, not prose.
3. **Fail-closed reachability.** Up front, before transforming, resolve the full chain `detected → CURRENT`. If any contract's detected version is below minimum, above current, or has no chained path, refuse with a named unsupported error and no write — extending the existing refusals in `migrateRunState` to a registry-level precondition and to the workflow-app contract.
4. **Round-trip / non-destruction prover.** A tool that runs the chain against a state file or app manifest and PROVES: (a) the result validates at `CURRENT`, (b) every source record/key survives into the output (append-only — nothing destroyed), (c) re-running migration is idempotent, (d) the original snapshot was not mutated (hash-before == hash-after). Emits a deterministic, fingerprinted `MigrationProof`.
5. **Workflow-app schema migration.** Bring the workflow-app schema under the same registry so an older `WORKFLOW_APP_SCHEMA_VERSION` app can be migrated forward (or fail closed with a precise reason) through the same declared mechanism, instead of being flatly rejected in `validateWorkflowApp`.
6. **Append-only proof storage.** Migration proofs and migrated snapshots persist beside the run/app as NEW files (never overwriting source); the report surfaces the verdict. Re-deriving from disk reproduces the same proof.
7. **Operator-facing verbs + report rendering.** `migration list` (the declared registry), `migration check <target>` (dry-run verdict + proof), `migration prove <target>` (round-trip proof), each on both surfaces with `--json` parity, and a workbench-renderable migration verdict panel.
