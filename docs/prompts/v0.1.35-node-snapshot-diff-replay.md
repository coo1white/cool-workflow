# DESIGN PHILOSOPHY — FREEBSD / UNIX DISCIPLINE

Each tenet is a hard constraint. The load-bearing one for this feature is flagged.

- **Mechanism, not policy.** The new module provides the *mechanism* (capture one node's derived state, fingerprint it, diff two, replay one). It decides nothing about *which* node matters — the caller names the node id. No heuristics about "interesting" nodes baked into the core.
- **Fail closed on source drift — the ambiguous→refusal rule. [LOAD-BEARING]** A node snapshot is DERIVED from `StateNode` + its referenced artifacts/evidence (like v0.1.25 summaries), carrying a `sourceFingerprint` (sha256 over the node's `id:status:updatedAt` plus referenced artifact/evidence ids+paths, mirroring the sha256 fingerprint pattern of `fingerprintStrings` in `src/state-explosion.ts` — note that helper is currently module-PRIVATE, so either replicate its pattern in the new module or export it additively first). On replay or diff, recompute the fingerprint from the current source: if it diverges, or the source node/artifact path is missing or unreadable, the operation REFUSES with a structured `PipelineContractError`-style error and a `stale|absent` freshness verdict — it never silently replays stale state or guesses. Equal-or-refuse; never a best-effort partial.
- **Reuse existing records/types, don't fork.** Operate on the real `StateNode` (`src/state-node.ts`) and the real `WorkflowRun.nodes` written by `writeRunNode`. Reuse the eval harness's `normalizeValue`/`stableStringify`/`lines` discipline from `src/multi-agent-eval.ts` for stable structural output — note these three helpers are currently module-PRIVATE in that file, so EXPORT them additively (no behavior change) and import them, rather than re-deriving them. Do NOT define a parallel node type, a second normalizer, or a competing replay engine.
- **CLI↔MCP parity via the capability-registry (v0.1.27).** Every new verb is declared once in `src/capability-registry.ts` as `surface: "both"` with matching `cli`/`mcp` bindings. `cw <cmd> --json` must be byte-identical to the `cw_<tool>` result (only an ISO `generatedAt`-style field may be now-derived, and the read-only payload probe strips it). Any surface-only or payload-divergent verb carries an explicit `reason` or the parity gate fails closed.
- **Backward compatible + additive.** No change to `StateNode`, `STATE_NODE_SCHEMA_VERSION`, the run-state schema, the pipeline contract, or existing eval-suite artifacts. New verbs and a new derived artifact directory only; pre-0.1.35 runs and snapshots stay loadable. (Exporting the previously-private eval-harness helpers and, if you choose, `fingerprintStrings` is purely additive and changes no behavior.)
- **Deterministic / replayable.** Replay injects `now` (no `new Date()` in the replay path — note `createStateNode`/`transitionStateNode`/`recordNodeError` stamp `new Date().toISOString()`, so the isolated replay must NOT call them with ambient time; it reconstructs from the snapshot with an injected clock). No ambient wall-clock, no filesystem ordering, no `Math.random` in the payload. Two replays of one snapshot are byte-identical; the diff of two snapshots is stable under key order and array order (sort like `lines()`).
- **Least astonishment.** Verb names mirror the existing `eval.*` family (`node.snapshot`, `node.diff`, `node.replay`) and the existing `node.*` family (`node.list`, `node.show`, `node.graph`) so an operator who learned either predicts these.
- **Not done until documented + tested (`.7.md` voice).** Ship `docs/node-snapshot-diff-replay.7.md` in the man-page voice of the existing `.7.md` set (e.g. `state-node.7.md`, `multi-agent-eval-replay-harness.7.md`, `release-tooling.7.md`), wire it into the docs index/README/CHANGELOG, and add a smoke test that runs green under `npm test`, `release:check`, and `version:sync`. Undocumented or untested = unshipped.

# THE FEATURE — Node Snapshot / Diff / Replay

**Current code reality.** The eval/replay harness in `src/multi-agent-eval.ts` works only at RUN/SUITE granularity: `createMultiAgentReplaySnapshot(run)` captures the *whole* run (topology, blackboard, workers, candidates, commits, trust, operator) into one `.cw/evals/<suite>/snapshot.json`, and `replayMultiAgentSnapshot` / `compareMultiAgentReplay` / `scoreMultiAgentReplay` / `gateMultiAgentEval` operate on that whole-run normalized blob. The harness's normalize/stable-output helpers (`normalizeValue`, `stableStringify`, `lines` — which strip timestamps/paths and sort arrays) currently exist but are module-PRIVATE; its public regression-finding shape is `{id, severity, category, reason, baselineRef, replayRef}` (`MultiAgentRegressionFinding`). The `StateNode` itself (`src/state-node.ts`) is the per-node record — created/transitioned/contract-checked and persisted one-file-per-node by `writeRunNode` into `<run>/nodes/<id>.json` — and `src/operator-ux.ts`'s `buildOperatorGraph` already renders nodes individually. But there is NO way to snapshot, fingerprint, diff, or replay a *single* node. The registry's `node.list`/`node.show`/`node.graph` (`src/capability-registry.ts`) are read-only views with no snapshot/diff/replay verbs, and the `eval.*` verbs (`eval.snapshot`/`replay`/`compare`/`score`/`gate`/`report`) have no per-node target.

**The gap.** Per-node snapshot/diff/replay: take one `StateNode` by id, derive a fingerprinted snapshot, diff two snapshots of the same node id structurally, and deterministically replay that one node in isolation — reusing the v0.1.23 harness's normalize/stable-stringify/compare discipline and the v0.1.25 fingerprint/freshness pattern, never forking `StateNode`.

Sub-capabilities:

1. **Node snapshot (derived + fingerprinted).** Given `<run-id> <node-id>`, derive a `NodeSnapshot` from the live `StateNode`: normalized inputs/outputs/artifacts/evidence/errors/parents/children/metadata (via the harness's `normalizeValue`, stripping timestamps/paths), plus a `sourceFingerprint` (sha256 over node `id:status:updatedAt` + referenced artifact ids/paths + evidence ids/paths). Persist under a new derived dir (e.g. `<run>/nodes/snapshots/<node-id>/<snapshot-id>.json`). The source `StateNode` JSON stays the truth.
2. **Fail-closed freshness.** Every load recomputes the fingerprint from current source and emits `valid|stale|absent`. `stale` (source mutated since capture) and `absent` (node or artifact path gone/unreadable) both REFUSE diff/replay with a structured error naming the divergence — never a silent stale replay.
3. **Structural node diff.** Given two snapshots of the SAME node id (or two ids explicitly), produce a stable, structural diff: per-section (inputs/outputs/artifacts/evidence/status/errors/links) `added|removed|changed|same`, sorted deterministically (same `stableStringify` ordering as the eval comparison sections). Byte-identical across repeated runs.
4. **Isolated deterministic replay.** Replay ONE node from its snapshot into an isolated workspace, with `now` injected (no ambient `new Date()`), reconstructing the normalized node state and re-validating it against its pipeline contract via the existing `state-node.ts` contract assertions (`assertNodeSatisfiesContract`) where applicable. Two replays of one snapshot are byte-identical.
5. **Replay-vs-source verdict.** Compare the isolated replay output to a fresh snapshot of the source node and emit a pass/fail verdict + findings, reusing the eval harness's finding shape (`severity/category/reason/baselineRef/replayRef`).
6. **Operator/Workbench rendering.** Render snapshot/diff/replay verdicts through `src/operator-ux.ts` (a panel + a `buildOperatorGraph` annotation marking which nodes have snapshots) and embed the canonical `--json` payload read-only in the v0.1.30 Workbench, adding nothing the CLI/MCP cannot show.
7. **Determinism guard.** A smoke assertion that snapshotting + replaying the same node twice yields byte-identical payloads and that mutating the source flips freshness to `stale` and blocks replay.
