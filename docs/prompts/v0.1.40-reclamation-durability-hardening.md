# PROVENANCE — THE FINDINGS THIS PROMPT CLOSES

The v0.1.39 self-audit ranked these REAL P1s (no P0). A **down-payment branch `fix/reclamation-durability-v0.1.40`** already lands P1-A, P1-B, P1-C, and P2-B; this prompt is the full release that also closes P1-D and P2-A.

- **P1-A — re-point outside the write-ahead boundary.** `freeBulk` re-pointed the result node in memory and deleted scratch, but `state.json` was persisted only AFTER (`run-registry.ts` `gcRun`), or never for direct callers. A crash in that window left a node referencing a freed path while the tombstone claimed capability unchanged. **[LANDED on the fix branch:** `prepareFree()` now re-points → DURABLY persists `state.json` (temp→fsync→rename) → PROVES no surviving node references a freed scratch path AND each re-pointed node's `loadNodeSnapshot === valid`, all BEFORE `freeBulk`; fail closed `repoint-incomplete`.**]**
- **P1-B — no "snapshot stays valid BEFORE freeing" assertion.** Re-point was best-effort and deleted regardless. **[LANDED:** `prepareFree()` asserts it; covered by the `repoint-incomplete` fail-closed path.**]**
- **P1-C — unlocked cross-process read-modify-write on `reclaimed.json`.** `commitTombstone` did load→push→write with no lock; a second reader for `prevTombstoneHash` widened the window; concurrent reclaimers could lose a tombstone (freed-without-proof). **[LANDED:** a per-run O_EXCL advisory lock (`withRunLock`, stale-steal) now wraps build+commit so the chain RMW is atomic.**]**
- **P1-D — `state.ts:writeJson` was non-atomic and unlocked (inherited).** **[LANDED in v0.1.40:** `writeJson` is now atomic (temp → rename) for EVERY caller, with `{ durable: true }` fsync for authoritative stores (state.json, registry overlays, scheduler store, reclaimed.json); a portable `withFileLock` (O_EXCL + stale-steal) serializes the home queue, archive overlay, repos registry, and reclamation chain read-modify-writes. See `docs/durable-state-and-locking.7.md`.**]**
- **P2-A — `validateSkeleton` was presence-of-keys, not content.** **[LANDED in v0.1.40:** `validateSkeletonAgainstRun` refuses (`skeleton-incomplete`) when extraction drops commits/evidence the run actually has; `validateSkeleton` also asserts the attestation chain / commits / evidence arrays are well-formed.**]**
- **P2-B — direct `runReclamation` callers had to `saveCheckpoint` manually.** **[LANDED:** the transaction now persists its own re-point durably.**]**

# DESIGN PHILOSOPHY — FREEBSD / UNIX DISCIPLINE

- **ORDER IS THE SAFETY PROPERTY [LOAD-BEARING].** Every state transition that frees or mutates authoritative bytes must be write-ahead: the proof/replacement lands durably BEFORE the destructive step. v0.1.39 established this for the tombstone; v0.1.40 must hold it for the re-point (done) and for EVERY `writeJson` (this release).
- **ATOMIC + DURABLE, EVERYWHERE [LOAD-BEARING].** Promote `writeJsonDurable` (temp → `fsync` → `rename` → dir `fsync`, already in `reclamation.ts`) into `state.ts` as the single persistence primitive. A crash/`ENOSPC` mid-write must leave EITHER the old file OR the new file — never a truncated one. No authoritative reader should ever brick on a half-written file.
- **SERIALIZE THE CROSS-PROCESS RMW [LOAD-BEARING].** The home queue, scheduler store, archive overlay, and `reclaimed.json` are read-modify-write across processes. A portable advisory lock (O_EXCL + stale-steal, as `withRunLock` does) must guard each, so a newly-added task can never vanish and a tombstone can never be lost.
- **FAIL CLOSED ON CONTENT, NOT JUST SHAPE.** `validateSkeleton` must assert the audit-essential CONTENT is present when the run has it (a run with commits must seal them), not merely that the key exists.
- **NO BEHAVIOUR CHANGE TO THE HAPPY PATH.** Atomicity/locking are invisible to correct single-writer use; pre-v0.1.40 runs load unchanged.

# THE FEATURE — Reclamation Durability Hardening

1. **Generalize the atomic durable write (closes P1-D).** Move `writeJsonDurable` into `state.ts` and make `writeJson` (or a new `writeJsonAtomic` that `saveCheckpoint`/registry/scheduler call) write temp → `fsync` → `rename` → best-effort dir `fsync`. Audit every authoritative writer (`saveCheckpoint`, `run-registry.ts` overlays, `scheduler.ts`, `daemon.ts` inbox, `commit.ts`, `coordinator.ts`) and route it through the atomic primitive. Reads stay pure.
2. **Lock the cross-process stores (extends P1-C).** Add a portable `withFileLock(path, fn)` (O_EXCL + stale-steal, generalized from `reclamation.ts:withRunLock`) and wrap the home queue (`run-registry.ts` queueAdd/queueDrain/saveQueue), the scheduler store, and the archive overlay read-modify-writes.
3. **Strengthen `validateSkeleton` to content (closes P2-A).** When `run.commits` is non-empty, require `skeleton.commits` non-empty; when any node/candidate/commit carries evidence, require `skeleton.evidenceDigests` non-empty; always require a populated `finalVerdict` and a non-empty `stateDigest`. Add the new failure detail to the `skeleton-incomplete` path.
4. **Promote the landed P1-A/B/C work into the release** (it is already on the fix branch): `prepareFree()` (re-point → durable persist → `repoint-incomplete` proof) and `withRunLock` around build+commit. Keep them; document them in the man-page.
